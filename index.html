              一次 llama_decode 的真实内部链路
              
        
              
          
              
            1. 校验输入
            
                
              
              • llama_decode(ctx, batch) 进入 。
                • 
              
              • 要求 token 和 embedding 二选一，且 n_tokens > 0。
                • 
              
              • 本次 trace 第一轮 batch_tokens=57，后续每轮 batch_tokens=1。
                • 
            
              
          
              
          
              
            2. batch -> ubatch
            
                
              
              • balloc->init() 把用户 batch 标准化：token、pos、seq_id、logits/output 标记。
                • 
              
              • memory->init_batch() 结合 KV cache 给出本轮可计算的 micro-batch。
                • 
              
              •  是真正进入计算图的更小单位，受 n_ubatch、上下文和 cache 空间限制。
                • 
            
              
          
              
          
              
            3. 准备 KV cache
            
                
              
              • memory_update(false) 处理延迟的 shift/copy。
                • 
              
              • prefill 会把 57 个 prompt token 的 K/V 写进 cache。
                • 
              
              • 生成阶段只把新 token 追加进 cache，历史上下文靠已有 K/V 复用。
                • 
            
              
          
              
          
              
            4. build/reuse graph
            
                
              
              • process_ubatch() 计算 graph 参数，能复用就复用上一张图。
                • 
              
              • 不能复用时调用 model.build_graph()，按模型架构选择 llm_build_llama。
                • 
              
              • 这一步只是搭出计算图节点，还没有真正算矩阵乘。
                • 
            
              
          
              
          
              
            5. set inputs + compute
            
                
              
              • res->set_inputs(&ubatch) 把 token、pos、attention mask、KV index 填进图输入张量。
                • 
              
              • graph_compute() 设置线程池和后端线程数。
                • 
              
              • ggml_backend_sched_graph_compute_async() 开始真正执行。
                • 
            
              
          
              
          
              
            6. 取回 logits
            
                
              
              • 图算完后拿到 res->get_logits()。
                • 
              
              • ggml_backend_tensor_get_async() 把需要输出的 logits 拷到 context 的输出缓冲。
                • 
              
              • server 后面才调用 sampler，把 logits 变成 token_id。
                • 
            
              
          
              
        
              

        
              
          Llama block 在图里具体算什么
          
              
            
              token -> embeddingggml_get_rows(tok_embd, tokens) 按 token id 查 embedding 行。
              
            
              RMSNorm 归一化当前 hidden state，为 attention 输入做尺度稳定。
              
            
              Q/K/V 投影 分别乘 wq/wk/wv 得到 query、key、value。
              
            
              RoPE 把位置信息旋进 Q/K，模型因此知道 token 的顺序。
              
            
              KV 写入 把本轮 K/V 存入 cache，后续 token 复用历史。
              
            
              attention常规路径是 K^T * Q -> mask/scale -> softmax -> V * score；开启 flash attention 时合成一个 op。
              
            
              输出投影 + 残差wo 把 attention 结果投回 hidden size，再和输入残差相加。
              
            
              FFN 做 gate/up、SiLU 激活、down projection，再走一次残差。
              
            
              lm_head最后 output_norm 加 model.output，得到 vocab 大小的 logits。
              
          
              
        
              

        
              
          
              prefill 为什么快一些第一轮一次处理 57 个 prompt token，矩阵乘可以批量跑，吞吐更好；它的核心成本是把整段 prompt 的 K/V 全部写进 cache。
              
          
              generation 为什么每步只喂 1 个 token采样出一个 token 后，下一轮只需要把这个新 token 接回上下文。历史 token 不重算 attention 投影，直接读 KV cache。
              
          
              “最底层”在哪里如果走 CPU，最终会到 ggml_compute_forward() 的 GGML_OP_MUL_MAT、GGML_OP_SOFT_MAX 等 case；如果走 GPU，同一张图会被 scheduler 切给对应 backend kernel。
